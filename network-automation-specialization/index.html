The Automation Mindset CORE CONCEPTS

Every network engineer eventually hits the same wall. You are managing 80 switches, a configuration change is required on all of them, and you find yourself opening 80 separate PuTTY sessions. This is not a workflow problem — it is a paradigm problem. The CLI was designed for a human operator, not a computer. The automation mindset begins with a single, uncomfortable truth: if a human has to be in the loop for a change to occur, it will never scale.

Why the CLI Was Never Meant to Scale

⛔ The Manual CLI Approach
  • 1 engineer × N devices = N2 effort
  • No diff / change tracking
  • Human typo risk on every session
  • Unstructured text output ("show run")
  • Hours-long maintenance windows
  • State is invisible until you SSH in
✅ The API-Driven Approach
  • 1 script × N devices = constant effort
  • Full change history in Git
  • Idempotent — safe to run repeatedly
  • Structured JSON/XML responses
  • Changes deployed in seconds
  • Operational state queryable programmatically

What Exactly Is an API? (The Network-Centric Definition)

An API (Application Programming Interface) is a formally defined contract between a software consumer and a software provider. When applied to a network device, it means: instead of you typing show ip interface brief and reading green text on a black screen, your Python script sends a structured request and receives a structured, machine-readable response.

🔬 The Analogy That Makes It Click

Think of a traditional restaurant with a rude, slow waiter (the CLI). You must speak in a specific dialect, wait for the waiter to be available, read handwritten scrawl back, and re-interpret it yourself. Now imagine a restaurant with a touchscreen kiosk (the API). You send a formally structured order (HTTP request), the kitchen processes it, and you receive a perfectly formatted receipt (JSON response) — no ambiguity, no interpretation, fully automatable.

The CLI is the waiter. The API is the kiosk. Both get food to the table — but only one can serve an infinite number of customers simultaneously.

The Three Generations of Network Management

evolution_of_netmgmt.txt
# ── GENERATION 1: Manual CLI (1990s–2010s) ─────────────────────
Tool : Telnet / SSH (PuTTY, SecureCRT)
Protocol: No protocol — raw terminal stream
Output : Unstructured text (requires human parsing)
Scale : 1–50 devices (human limit)

# ── GENERATION 2: SNMP & Screen-Scraping (2000s–2015s) ─────────
Tool : Expect scripts, Netmiko, SNMP managers
Protocol: SNMP (UDP) for polling, SSH for config
Output : OID trees (SNMP) / scraped text (SSH)
Scale : 50–200 devices (fragile, brittle)

# ── GENERATION 3: API-Driven / Model-Driven (2015–Present) ──────
Tool : REST APIs, NETCONF, RESTCONF, gRPC
Protocol: HTTP/HTTPS (REST), SSH (NETCONF)
Output : Structured JSON / XML / YANG-validated data
Scale : Unlimited (idempotent, version-controlled)

REST APIs: The Language of Modern Network Automation

A REST API (Representational State Transfer) communicates via standard HTTP verbs. As a networker, you already understand the concept — you just use different vocabulary:

HTTP Verb Network Equivalent What It Does
GETshow running-configRead / retrieve current state
POSTconfigure terminal + new itemCreate a new resource
PUT / PATCHinterface config modificationUpdate an existing resource
DELETEno interface / no ip routeRemove a resource
💡

Key Insight: Every time a Python script or Ansible playbook "talks to" a Cisco, Juniper, or Arista device in a modern environment, it is almost certainly using one of these four HTTP verbs over an API. The device's operating system has been extended with an HTTP listener — a small web server — specifically to receive these requests. This is what makes automation possible at scale.

Linux: The Home of Automation TECHNICAL DEEP DIVE

When people say "network automation," they almost invariably mean "running code on a Linux host." This is not a coincidence — it is an architectural truth. Python, Ansible, Git, Docker, Terraform, and virtually every automation tool in the NetDevOps stack was either written for Linux, runs best on Linux, or is embedded within Linux. Understanding why requires a brief look at the OS's design philosophy.

The Unix Philosophy: Small Tools That Compose

"Write programs that do one thing and do it well. Write programs that work together. Write programs that handle text streams, because that is a universal interface."

— Doug McIlroy, Bell Labs, 1978 · The principle behind every automation pipeline you will ever build.

This philosophy means that in Linux, you routinely chain commands together using the | (pipe) operator. Each small tool transforms data and passes it to the next — exactly how an automation pipeline works:

linux_pipeline_example.sh
# Real-world example: Parse only BGP neighbor IPs from a config backup
$ cat router_config.txt | grep "neighbor" | awk '{print $2}' | sort -u
10.0.0.1
10.0.0.2
192.168.1.254

# Decomposed: cat (read) → grep (filter) → awk (transform) → sort (deduplicate)
# Each tool is <10KB. Together they replace a 200-line Excel macro.

Linux Filesystem vs. Cisco IOS: A Structural Comparison

If you have worked on Cisco devices, you understand a filesystem rooted in flash: or bootflash:. Linux uses a radically different, unified hierarchy. Everything — including hardware devices — is represented as a file.

Cisco IOS Filesystem
  • flash:/running-config
  • flash:/startup-config
  • flash:/c2900-universalk9.bin
  • nvram:/startup-config
  • No user home directory
  • No package manager
Linux Filesystem (FHS)
  • /etc/ — System configuration files
  • /home/ — Per-user workspaces
  • /usr/bin/ — Installed programs
  • /var/log/ — Log files
  • /tmp/ — Temporary files
  • /dev/ — Hardware as files

The Linux Filesystem Hierarchy Standard (FHS)

Every Linux distribution follows a standard layout. As a NetDevOps engineer, these are the directories you will interact with most:

/                         # Root of the entire system — like "C:\" on Windows
├── etc/                    # System-wide configs (SSH, network, Ansible)
│   ├── hosts              # Static DNS resolution (/etc/hosts)
│   └── ansible/          # Global Ansible configuration (Week 6)
├── home/                   # All user home directories live here
│   └── netdevops/        # YOUR workspace — scripts, playbooks, etc.
├── usr/
│   ├── bin/                # User-level executables (python3, ansible, git)
│   └── local/             # Locally compiled/installed software
├── var/
│   └── log/               # System and application logs — critical for debugging
├── tmp/                    # Temporary files — wiped on reboot
└── dev/                    # Hardware devices represented as files
⚠️

The Networker's "Aha" Moment: In IOS, configuration is stateful and immediate — you type it and it's live. In Linux, configuration lives in text files inside /etc/. This is why infrastructure automation is so powerful — you edit files programmatically (with Ansible, Python, or Jinja2 templates) and then apply them. The entire configuration of a device can be stored, versioned, and deployed from Git.

Essential Linux Commands for the Network Automator

Master these 15 commands first. They will appear in every lab, every script, and every Ansible task you write from Week 2 onward.

essential_commands.sh
# ── NAVIGATION ──────────────────────────────────────────────────
$ pwd                   # Print Working Directory — "where am I right now?"
/home/netdevops

$ ls -la                # List ALL files (including hidden) with permissions
drwxr-xr-x netdevops scripts/
-rw-r--r-- netdevops inventory.yaml

$ cd /home/netdevops/scripts  # Change Directory (absolute path)
$ cd ..                 # Go up one directory level
$ cd ~                  # Return to your home directory (shortcut)

# ── FILE OPERATIONS ─────────────────────────────────────────────
$ mkdir -p projects/week1   # Make directory + parent dirs if needed
$ touch notes.txt          # Create an empty file
$ cp notes.txt backup.txt   # Copy a file
$ mv backup.txt archive/    # Move / rename a file
$ rm -rf old_configs/      # DANGER: Remove dir recursively — no recycle bin!

# ── READING FILES ───────────────────────────────────────────────
$ cat router_config.txt    # Print entire file to screen
$ less router_config.txt    # Page through a file (q to quit)
$ grep "interface" config.txt# Search for pattern in a file

# ── PERMISSIONS (critical for scripts!) ─────────────────────────
$ chmod +x my_script.sh    # Make a script executable
$ ./my_script.sh           # Execute the script from current directory

# ── NETWORKING UTILS (familiar territory!) ──────────────────────
$ ping -c 4 8.8.8.8        # Send exactly 4 ICMP packets
$ ip addr show             # Show interface IPs (modern replacement for ifconfig)
$ curl -s https://api.ipify.org # Make an HTTP GET request — your first API call!
📘

Pro Tip — The man Pages: Every command has built-in documentation. Type man ls, man grep, or man curl to read the manual for any command. Think of it as ? in Cisco IOS — but comprehensive. Press q to exit.

Check for Understanding ASSESSMENT

The following questions are written to mirror a professional NetDevOps Associate certification environment. Each question has one correct answer. Click an option to reveal the explanation after selecting.

// Question 01 of 03 · Conceptual
A network engineer needs to retrieve the operational state of all BGP neighbors on 200 routers simultaneously and store the results in a structured format for automated analysis. Which approach is MOST appropriate for this use case?
ASSH into each router sequentially using a Python loop with Paramiko and parse the output of show bgp summary using regular expressions.
BUse a RESTCONF API call with an HTTP GET request to query the BGP operational state YANG model on all devices simultaneously, receiving structured JSON responses.
CConfigure SNMP v2c polling on all devices and use a MIB browser to collect BGP neighbor status OIDs.
DWrite an Expect script to automate SSH connections and extract neighbor state from CLI output using string matching.
✅ B is correct. RESTCONF with a YANG-modeled data source provides structured JSON output natively — no parsing required. Options A and D are "Generation 2" screen-scraping approaches: functional but fragile, as any output format change breaks the parser. Option C (SNMP v2c) lacks the depth and writability of modern APIs, and MIB-based data is not JSON-structured. The key indicator in the question was "structured format for automated analysis" — this points directly to an API-based solution.
// Question 02 of 03 · Linux Practical
A NetDevOps engineer creates a Python automation script called deploy_vlans.py on a Linux host. When attempting to run it with ./deploy_vlans.py, the system returns: -bash: ./deploy_vlans.py: Permission denied. What is the MOST LIKELY cause and the correct remediation?
AThe Python interpreter is not installed. Run sudo apt install python3 to install it.
BThe file is in the wrong directory. Move it to /usr/bin/ using sudo mv deploy_vlans.py /usr/bin/.
CThe file is missing the execute permission bit. Run chmod +x deploy_vlans.py to grant execute permission, then re-run.
DThe script must be run as root. Prefix the command with sudo.
✅ C is correct. In Linux, newly created files do not have the execute bit set by default — this is a security feature. The chmod +x command adds the execute permission. You can verify the change by running ls -la deploy_vlans.py — the permissions should change from -rw-r--r-- to -rwxr-xr-x. Option D (sudo) would only be needed if the script requires root-level access to system resources — unrelated to this error. Understanding file permissions is fundamental; they control every executable, every Ansible playbook, and every SSH key on your system.
// Question 03 of 03 · Architecture
Which Linux directory is the CORRECT location to store system-wide configuration files for an automation tool like Ansible that should apply to all users on the host? (Referencing the Filesystem Hierarchy Standard)
A/tmp/ — Temporary files used by running processes
B/home/user/.config/ — User-specific application configuration
C/etc/ — System-wide editable configuration files for all users
D/usr/bin/ — User-accessible system executable binaries
✅ C is correct. Per the Filesystem Hierarchy Standard (FHS), /etc/ is the designated location for host-specific system configuration. The global Ansible configuration file lives at /etc/ansible/ansible.cfg. Option A (/tmp/) is wiped on reboot — never store configuration there. Option B is correct for user-specific config but would not apply to all users. Option D (/usr/bin/) is for executable programs, not configuration files. Understanding the FHS is the difference between a clean, maintainable system and an "it works on my machine" disaster.
